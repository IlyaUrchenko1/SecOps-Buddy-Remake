# MITRE ATT&CK T1030 --- Ограничения размера передачи данных (Data Transfer Size Limits)

## Источник

-   Техника MITRE ATT&CK: https://attack.mitre.org/techniques/T1030/

------------------------------------------------------------------------

## Обзор

**T1030 (Ограничения размера передачи данных)** — это техника эксфильтрации, при которой злоумышленник намеренно ограничивает размер отдельных передач данных.

Вместо того чтобы выводить большой объём данных за одну сессию, атакующий дробит данные на множество небольших передач. Такой подход помогает избежать детектирования со стороны систем безопасности, отслеживающих аномальные объёмы исходящего трафика или крупные загрузки.

Пример:

Эксфильтрация 1 ГБ данных

Обычная передача: одна загрузка по 1 ГБ

Поведение по T1030: 10 000 загрузок по 100 КБ

С точки зрения мониторинга это выглядит как множество коротких сетевых соединений, а не одно большое.

------------------------------------------------------------------------

## Типовые сценарии использования

Эта техника может встречаться в ряде вариантов эксфильтрации:

-   Передача по каналам Command and Control (C2)
-   DNS-туннелирование
-   Постепенная загрузка через HTTP/HTTPS
-   Использование облачных API
-   Пользовательские C2-протоколы

Связанные тактики MITRE:

-   Экcфильтрация: https://attack.mitre.org/tactics/TA0010/

## Наблюдаемое поведение

Распространённые индикаторы этой техники включают:

### Многоразовые короткие соединения

Процесс многократно соединяется с одним и тем же удалённым хостом.

Пример шаблона:

process → remote_ip\
process → remote_ip\
process → remote_ip

### Изменяющиеся локальные эфемерные порты

Каждое соединение использует новый локальный порт.

Пример:

49152 → 185.x.x.x\
49160 → 185.x.x.x\
49168 → 185.x.x.x

### Небольшие многократные передачи

Каждое соединение передаёт небольшой объём данных.

## Детектирование в SecOps Buddy

Текущее решение выявляет эту технику поведенческим анализом сетевых соединений.

Данные собираются из стандартных сетевых утилит:

    ss
    netstat

Система периодически делает снимки активных соединений и анализирует повторяющиеся паттерны.

Соединения группируются по такому ключу:

    (remote_ip, remote_port, protocol, process_name)

Для каждой группы детектор оценивает:

-   частоту повторяющихся соединений
-   количество различных локальных эфемерных портов
-   повторяемость одного и того же удалённого адреса
-   вовлечённость одного и того же процесса

Безопасные сценарии фильтруются по возможности:

-   трафик на localhost
-   приватные диапазоны сетей
-   известные системные процессы

Оповещение срабатывает, если один процесс многократно создаёт короткие соединения с одним удалённым адресом.

------------------------------------------------------------------------

## Ограничения

Текущий способ анализирует только состояние сокетов и поведение соединений.

Он **не** измеряет:

-   точный объём переданных данных
-   содержимое пакетов
-   фактический объём передачи

В связи с этим детектирование основывается на поведенческих признаках, а не на прямом измерении объёма данных.

## Возможные улучшения

### Инспекция пакетов

С использованием инструментов захвата пакетов:

-   pcap
-   tcpdump
-   libpcap

Это позволило бы измерять фактический объём переданных данных.

### Анализ сетевых потоков

Мониторинговые системы потоков:

-   NetFlow
-   IPFIX
-   Zeek

Они дают информацию о:

-   объёме передачи
-   длительности сессии
-   частоте трафика

### Телеметрия через eBPF

eBPF позволяет получить более глубокую видимость сетевой активности процессов:

-   системные вызовы send / recv
-   подсчёт переданных байт
-   сетевые события на уровне процессов
